# Atome / eVe Design Map

Status: Initial design map after the Atome open / eVe closed boundary validation.

Purpose:

- Map the design system even though the maintained UI is generated by JavaScript, not static HTML/CSS.
- Identify ownership, token sources, style injection points, visual factories, and generated CSS exceptions.
- Prevent design drift across eVe panels, tools, Matrix, Molecule, MTraX, and Atome framework components.

Mandatory Use:

- Before changing visual behavior, consult this file, then verify the owning source module.
- Prefer existing tokens, presets, and visual factories over new inline style islands.
- Do not add product design logic to Atome open modules unless it is product-neutral.
- Do not treat generated or vendored CSS files as the source of truth for eVe product design.

## Mandatory Pre-Implementation Gate

No visual, UI, panel, tool, Matrix, Molecule/MTraX, asset, style, token, preset, JavaScript-generated CSS, or visual factory change may start until the relevant maps have been consulted:

- Use `maps/CODEMAP.md` to locate the owning module and avoid creating duplicate visual systems.
- Use `maps/API_MAP.md` when the visual change exposes, consumes, or changes runtime APIs, globals, tool capabilities, or panel APIs.
- Use this file to identify the owning token source, style generator, factory, asset class, or generated CSS exception.
- Use `maps/ARCHITECTURE_MAP.md` when the visual change crosses Atome/eVe ownership, layer contracts, lifecycle, rendering, command flow, or runtime mode boundaries.

Implementation may proceed only after the design owner, existing reusable tokens or factories, boundary status, and required map updates are known.

## Design Model

Atome/eVe does not use a classic static HTML plus authored CSS architecture for product UI. The maintained product UI is created at runtime through JavaScript:

- DOM nodes are created by JavaScript factories.
- Style objects are passed through factory APIs or applied directly to elements.
- Shared tokens are exposed as JavaScript constants and CSS custom properties.
- Some modules inject `<style>` tags from JavaScript strings or generated rule objects.
- Static CSS files exist, but they are framework, vendor, or build artifacts rather than the main eVe design source.
- DOM `data-*` attributes are not a design or model source of truth. Product renderers may use them only for short roles, view ids, renderer hints, and transient UI state; visual caches, media sources, media identifiers, group timelines, and serialized model data belong to structured JavaScript state, stores, or renderer caches.
- `AtomGraph` visual, semantic, and focus order fields are structural/navigation projection data, not design tokens or product style rules. Design code must not infer product styling ownership from these fields; renderer and design surfaces remain responsible for visual tokens and paint output.
- `AccessibilityGraph` and `AccessibilityBridgeProjection` are semantic/navigation contracts, not visual style contracts. Product design code may consume their labels, roles, orders, and actions for accessible presentation, but must not treat bridge nodes as DOM ownership, CSS selectors, ARIA source-of-truth, or per-Atome visual styling state.
- Tool palette child lists are renderer behavior state, not design metadata; `data-tool-children` and `data-source-tool-children` must stay absent from product DOM output.
- Media mount errors are runtime diagnostics, not DOM design state; visual surfaces may reflect media readiness, but `data-media-api-error` must not persist in exported product DOM.
- Canvas surfaces are dense renderer outputs and must expose a short `data-role` in maintained DOM exports so active surfaces can be counted, pooled, and audited.

## Ownership Boundary

### Atome Open Design Surface

Path: `atome/`

Role: Product-neutral assets, generic Squirrel component builders, framework CSS artifacts, and bundled vendor CSS.

Allowed responsibilities:

- Generic icons, logos, and reusable assets under `atome/src/assets/`.
- Generic Squirrel component builders under `atome/src/squirrel/components/`.
- Product-neutral Squirrel table templates and table style variants in `atome/src/squirrel/components/table_visual_contract.js`.
- Canonical open system-control builders for Button, Slider, Input, and Console when the control contract is product-neutral.
- Framework-level CSS artifacts such as `atome/src/css/squirrel.css`.
- Vendored library CSS such as `atome/src/js/event-calendar.css` and `atome/src/js/leaflet.min.css`.

Forbidden responsibilities:

- eVe product chrome, panels, private tool styling, Matrix styling, MTraX/Molecule product styling, or branding composition.

### eVe Closed Design Surface

Path: `eVe/`

Role: Closed product design system, UI tokens, panel chrome, tool visuals, Matrix visuals, runtime style injection, and product interaction visuals.

Allowed responsibilities:

- Product tokens and CSS variable injection.
- Panel, dialog, footer, menu, ribbon, flower, Matrix, tool, Molecule, and MTraX visual systems.
- Product-specific style strings generated by JavaScript.
- Closed design composition and product interaction styling.

Forbidden responsibilities:

- Generic Atome framework style rules that must be reusable outside eVe.

## Explicit Open / Closed Design Contract

Design ownership follows product specificity:

- Atome open design may contain only product-neutral Squirrel component structure, reusable framework visual assets, shell-level CSS, vendor CSS, and generated framework artifacts.
- eVe closed design owns product chrome, panels, menu, ribbon, flower, Matrix, Finder-facing product UI, private tool visuals, Molecule/MTraX visuals, branding, and product-specific Atome object rendering.
- Static CSS files are not the source of truth for eVe product design. eVe product visuals must be driven by JavaScript tokens, factories, structured style objects, and documented generated style exceptions.
- Generated or vendored CSS may be referenced as an exception only when this map records its ownership and purpose.

Cross-boundary design rules:

- Atome modules must not import or depend on eVe product tokens, panel chrome, product factories, branding, or closed style generators.
- eVe design modules may consume Atome product-neutral assets or Squirrel component contracts, but must not redefine generic framework styling as closed product-only behavior.
- Product object styling for Atome instances inside eVe remains closed design behavior unless a reusable rendering contract is explicitly promoted to Atome.
- Any promotion of eVe visual behavior into Atome requires a product-neutral API, reusable token contract, tests or visual verification, and updates to `maps/CODEMAP.md`, `maps/API_MAP.md`, and `maps/ARCHITECTURE_MAP.md`.

The generated transparent shadow sprite opts out of Bevy automatic batching to keep Safari/WebGPU first-frame presentation deterministic. Each card uses a slight `+3%` same-hue lift from its visual lane entête fill for legibility, while the lane body is derived from that entête with `laneShadePercent: -10`. Entête rectangles are pixel-snapped flush to the handedness edge. The dashboard lateral entête shadow is a single external `header_side_shadow` image/texture gradient placed outside the entête rectangle, never a shape shadow or overlay over the color block. The dashboard depth over the project is represented by one low-layer translucent project veil, one high-layer violet `reserved_band_fill` mask in the toolbox-reserved band below the dashboard, and one external vertical `bottom_shadow` texture gradient that starts at the dashboard bottom and may spill into that band; it is not a second renderer or DOM backdrop filter. SVG data-url sources for icons, shadows, and label backdrops live in `dashboard_svg_sources.js`; they are static projection assets, not CSS or a parallel UI renderer. The `backdrop_blur` path is not active by default.
- The Bevy dashboard follows the R&D canvas mockup structure: joined lanes derive a darker body color from their entête in the unfocused overview, square/double-width cells based on lane height, shadowed internal inset cards for real items only, white bottom-left item labels rendered through a single higher-density `text_style` texture with a lighter stroke, diffuse shadow, and opt-in shrink-to-fit before clipping, one lateral entête shadow, one bottom shadow, and one high-density texture/icon record per header. Calendar cards project a compact start-date display while label editing keeps the full start value. It opens with no focused category; after an entête click, the active category color spreads for `transitions.categoryFocusMs` (`500ms` by default) from the clicked lane through two temporary Bevy shape records, one under card content and one under header icons/text, then settles with the dashboard background, table, every lane, and every entête as one uniform focused backdrop while inactive header text/icons remain visibly dimmed. Clicking the active entête reverses the same Bevy shape records from full dashboard coverage back to that lane before restoring the unfocused multi-color overview. Switching from one active entête to another keeps the previous color as the full-dashboard base, fades the new source lane in first, then expands it across the dashboard so no new color appears abruptly. The active category's deduplicated item list is still distributed once across the existing visual lanes without data duplication. Dashboard focused mode does not reserve or render a `+` rail; new item creation is no longer owned by the entête click surface.
- Dashboard header/icon/label/media texture sharpness and reuse are owned by the Bevy media texture resolver plus `bevy_media_texture_cache.js`, while final canvas sharpness is owned by the browser Bevy DPR-aware surface contract. Header SVG icons and detailed labels keep the shared high-density `texture_scale: 4` contract when their logical display size is small; project previews and contact faces use the shared media `texture_scale` contract at `max(2, devicePixelRatio)` so card thumbnails stay sharp without 4x upload cost. Repeated dashboard opens, scrolls, or rubrique changes must reuse unchanged text, SVG, media raster payloads, and stable item records by content/rich-text edit state/style/dimensions/DPR/texture scale instead of adding per-card canvases, DOM text, pre-scaled Dashboard geometry, dashboard-only texture systems, or empty delete/recreate frames for cached content.

- `tests/eve/dashboard_render_performance_contract.test.mjs` guards dashboard record budgets, bounded visible layout, stable category+item record ids, exact lane snap points, flush pixel-snapped entêtes, square centered-shadow card interiors through `material.shadow`, warmup/opening race prevention, compact high-density icon projection, one-record high-density text labels with opt-in shrink fitting, single external texture-gradient `header_side_shadow` with no `material.shadow`, one project veil, one violet reserved-band fill, one external bottom texture shadow with no `material.shadow`, no default `backdrop_blur` emission, cache-first header activation, texture cache reuse, and animation-frame render coalescing. `tests/eve/dashboard_focus_color_contract.test.mjs` guards the focused-category uniform backdrop, entête-family card colors, overview lane darkening, and unfocused multi-color restoration contract.

- Dashboard layout is expected to stay flush with the maintoolbox while open. Toolbox reservation is based on painted toolbox controls/surfaces, not transparent ribbon wrapper containers, so the dashboard bottom aligns to the first visible toolbox pixel. Dashboard rubrique visibility is a profile preference filtered before canvas layout/rendering; it must not create DOM dashboard cards, DOM state mirrors, or CSS dashboard visibility layers.
- Lane drag scroll keeps per-lane offsets in dashboard runtime state, requests Bevy updates through one animation-frame scheduler during pointer/wheel/inertia movement, projects the final offset from pointer velocity, then animates to the nearest exact snap point anchored to the entête side. Entête press gestures activate a rubrique only on release without movement; vertical drags over entêtes scroll the rubrique stack and must not open a rubrique.
- Dashboard open and close fade through Dashboard Bevy record opacity, never through CSS opacity on the canvas or the neutral workspace host. Opening projects the Dashboard records into the neutral `__eve_dashboard_workspace__` scene, claims the shared project canvas foreground for that neutral scene, starts from record opacity `0`, and fades records to their resolved visual opacity without loading a user project. Calendar cards display the event name and start date as two separate editable text records, and the Calendar panel must refresh its event list on reopen so the panel text matches dashboard commits. Contacts are sorted alphabetically for Dashboard projection, with the current user pinned to the first visible contact card when available; local `eve_contacts_local` payloads still replace matching directory entries and remain the only editable contact cards. Non-local contact cards keep their normal click behavior without creating dashboard aliases. No dashboard DOM input, item card, visible text node, CSS editor, or per-item overlay is allowed.
- Dashboard card media stays canvas-rendered. Contacts with `metadata.user_face` project a single Bevy image record above the card color and below the name text, carrying `media_fit: "cover"`, `texture_scale: max(2, devicePixelRatio)`, and the exact card content radius so the photo fills the card without deformation and is alpha-clipped to the rounded card. Projects with `metadata.project_preview_source` project a single Bevy image record above the card color and below the name text, carrying `media_fit: "contain"`, `texture_scale: max(2, devicePixelRatio)`, persisted `media_width`/`media_height`, and the same radius so the full miniature keeps its source ratio, the card color fills any margin, and image pixels cannot bleed into rounded corners. Image-backed cards add a bottom label backdrop above the media and below the text: a Bevy image record generated from a bottom-only-rounded SVG path, full card-content width, bounded to a readable `24px` logical minimum and `42px` cap, same card fill color, and `0.6` opacity; their title text record must use that exact rectangle with `baseline: "middle"` and `padding_y: 0`, while non-media cards and calendar secondary fields keep their existing text geometry. Product-neutral image texture fitting, per-node media texture scale, cache separation, and rounded clipping are owned by `bevy_media_texture_image_fit.js` / `bevy_media_texture_resolver.js`, while this label backdrop is owned by `dashboard_records.js`; neither path may use CSS, DOM overflow, visible DOM cards, or Dashboard-local renderer fallbacks. Project media comes from a renderer-owned Bevy/WebGPU PNG capture produced in the reusable hidden preview iframe/canvas at a default `640x400` source size, not from a live embedded project in the Dashboard block; default Dashboard boot must use only persisted preview metadata and must not load projects or cold-capture previews in the background before a project click. The capture surface must stay paintable for WebGPU while non-interactive and behind the app, wait for deferred Bevy resources, and reject transparent or nearly empty PNGs instead of visually accepting a blank card. These visuals must not use CSS backgrounds, DOM cards, per-item canvases, Matrix DOM clones, DOM screenshots, icons as substitutes, or renderer fallbacks.

- Maintained DOM projection fixtures are audited and validated by `scripts/check_dom_projection_guardrails.mjs`, including measurable DOM size, element/tag counts, data attribute counts, inline style ratio, inline handlers, data URI/base64/preview-signature counts, local source attribute leaks, duplicate ids, document roots, media errors, and canvas/video counts so dense visual surfaces remain renderer-owned rather than DOM-owned. Full-app debug captures generated by `scripts/export_dom_subtrees.mjs` remain `.snapshot` diagnostics under `temp/`; versioned `.dom` examples are maintained subtree projections only.
- Video recordings made from the capture tool snapshot the live preview through `eVe/intuition/shared/capture_video_poster_runtime.js` before the preview is disposed, then mount and persist that image on the reused `video_recording_*` project atome returned by the UI stop path. The later media-file poster generator must not overwrite a poster already mounted on that host.
- Project-visible video molecule previews are finalized by `eVe/intuition/shared/group_video_poster_runtime.js`, which captures and persists an image poster before refreshing the group visual instead of leaving a live video surface in the atome square.
- MTraX clip preview metadata renders audio waveforms and video thumbnails from the clip source window before visual layout. Split and crop operations must display the `in`/`out` media interval without restarting from the source beginning or stretching the full-source waveform/thumbnails into the cropped clip width.
- During active MTraX clip crop gestures, the clip preview is temporarily kept at its pre-gesture width and clipped by the body overflow so the cropped waveform or thumbnails are hidden live without resampling; the final source-window redraw runs after the gesture ends.

Design rule: MTraX/Molecule visuals are closed product workflow design unless promoted through a deliberate Atome media contract.

## Login Sequence Surface

Primary sources: `eVe/intuition/tools/user_login_sequence.js`, `eVe/intuition/tools/user_login_choice.js`, `eVe/intuition/tools/user_login_credentials.js`, `eVe/intuition/tools/user_login_mirrored_text.js`, `eVe/intuition/tools/user_login_choreography.js`

Role:

- Owns the unauthenticated initial login visual surface. The first surface is a full-screen two-zone choice on the textured login shell: the upper area enters without an account on a dark mauve-to-black animated gradient, the lower area opens authentication on a deep violet-to-black animated gradient, and the single persistent white Atome logo starts exactly on the junction as an authentication control with a soft pulsing violet/white halo.
- The initial surface fades in directly over the body over about four seconds. The persistent logo is mounted at full opacity from the start of the accueil surface and the real logo image stays white through the fixed `brightness(0) invert(1)` filter; the entry fade must not animate the logo wrapper or image opacity. `eve:login-choice-mounted` is emitted as soon as the choice surface and logo are mounted so hidden warmups can start without waiting for the long fade, while `eve:login-choice-visible` is emitted only after the entry fade has finished. Workspace/dashboard/toolbox visuals may be prepared behind this login surface, but the login surface must not fade out until the workspace path has the project canvas, dashboard runtime, dashboard records present in the Bevy project scene, and main ribbon ready to reveal. The login exit fade for guest/desktop reveal is about two seconds; the main ribbon must not reveal before dashboard records are ready, otherwise the user sees toolbox chrome before the final dashboard scene.
- The credential surface is a full-screen 1/7-5/7-1/7 composition for phone, OTP, and password. The upper band owns the current instruction or OTP error, the central band owns native input capture plus mirrored value display and a mirrored blinking caret, and the persistent Atome logo moves to the bottom band as the validation control. The native input is visually hidden so password bullets and caret are rendered only by the clean mirrored text, and successful password submit clears native focus/selection before reveal. In test/demo OTP mode the upper band displays `O.T.P: {code}` instead of the production SMS instruction.
- On the password step, clicking the persistent Atome logo with an empty password returns to the phone step in the credential surface, restores phone focus and the mirrored caret, and does not submit credentials; clicking it with a non-empty password keeps the existing password validation path. Every credential step display resets the instruction and mirrored typed text opacity, blur filter, and transform, and restores caret opacity while open so residual animation state cannot leak across phone, OTP, password, logout, or empty-password returns.
- Logout reentry must reset any existing shared login sequence to the first choice surface before `Connexion / inscription` opens credentials again, so stale phone/OTP/password surfaces are never preserved across session teardown.
- `user_login_choreography.js` owns the continuous animation contract: no duplicated login logo, no teleportation, resize recentering for the current target, scroll-down central swaps between phone/OTP/password, fade/micro-motion text transitions, living top/bottom bands, invalid pulses, animated return to the first screen, cleanup of its own exit fade handles, and sequenced desktop reveal for password success and guest entry. After local password-form validation but before `AdoleAPI.auth.bootstrap(...)`, the single persistent logo starts the fast centered wait motion and `user_login_session_opening_view.js` shows the neutral localized `Validation en cours / merci de patienter` message. After real password validation, the same view switches to `Bienvenue, / nous ouvrons votre session`; this success animation is not awaited by the workspace/profile/project flow. The final reveal uses the fast authenticated timing profile for middle fade, visible text fade, band exit, and docking to `button[data-role="eve_intuitionx-handle"]`.
- Reuses eVe design factories for text, phone input, and password input. `eVe/intuition/tools/user_login_visual_contract.js` owns the login-only structured visual tokens, the `LOGIN_GRADIENT_MOTION` speed/intensity parameters for animated gradient breathing, the `LOGIN_TEXT_STYLE` typography scale/opacities, band geometry, textured shell, and native-input capture styles for this sequence.
- Uses `eVe/i18n/languages.js` keys under `eve.user.login_choice.*` and `eve.user.login_sequence.*` for visible and assistive text.

Design rule: this surface must remain minimal and must not reintroduce the legacy compact phone/password panel as the initial unauthenticated login UI. It must not draw a phone mockup, notch, extra buttons, error text, step indicators, or a circle/box/background behind the logo. The active flow is choice -> phone -> OTP -> password -> final reveal.

## Validation Expectations

For any design change:

- Verify the owning JavaScript visual module directly.
- Check whether the change should be tokenized in `system_ui_tokens.js`, `eVe_look.js`, or a feature visual token module.
- Verify touched file size and avoid expanding oversized visual/runtime files.
- Run the narrowest relevant visual/runtime test when one exists.
- For significant UI changes, use browser or probe verification on the real rendered surface.
