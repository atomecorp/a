# Atome / eVe Design Map

Status: Initial design map after the Atome open / eVe closed boundary validation.

Purpose:
- Map the design system even though the maintained UI is generated by JavaScript, not static HTML/CSS.
- Identify ownership, token sources, style injection points, visual factories, and generated CSS exceptions.
- Prevent design drift across eVe panels, tools, Matrix, Molecule, MTraX, and Atome framework components.

Mandatory Use:
- Before changing visual behavior, consult this file, then verify the owning source module.
- Prefer existing tokens, presets, and visual factories over new inline style islands.
- Do not add product design logic to Atome open modules unless it is product-neutral.
- Do not treat generated or vendored CSS files as the source of truth for eVe product design.

## Design Model

Atome/eVe does not use a classic static HTML plus authored CSS architecture for product UI. The maintained product UI is created at runtime through JavaScript:
- DOM nodes are created by JavaScript factories.
- Style objects are passed through factory APIs or applied directly to elements.
- Shared tokens are exposed as JavaScript constants and CSS custom properties.
- Some modules inject `<style>` tags from JavaScript strings or generated rule objects.
- Static CSS files exist, but they are framework, vendor, or build artifacts rather than the main eVe design source.

## Ownership Boundary

### Atome Open Design Surface

Path: `atome/`

Role: Product-neutral assets, generic Squirrel component builders, framework CSS artifacts, and bundled vendor CSS.

Allowed responsibilities:
- Generic icons, logos, and reusable assets under `atome/src/assets/`.
- Generic Squirrel component builders under `atome/src/squirrel/components/`.
- Framework-level CSS artifacts such as `atome/src/css/squirrel.css`.
- Vendored library CSS such as `atome/src/js/event-calendar.css` and `atome/src/js/leaflet.min.css`.

Forbidden responsibilities:
- eVe product chrome, panels, private tool styling, Matrix styling, MTraX/Molecule product styling, or branding composition.

### eVe Closed Design Surface

Path: `eVe/`

Role: Closed product design system, UI tokens, panel chrome, tool visuals, Matrix visuals, runtime style injection, and product interaction visuals.

Allowed responsibilities:
- Product tokens and CSS variable injection.
- Panel, dialog, footer, menu, ribbon, flower, Matrix, tool, Molecule, and MTraX visual systems.
- Product-specific style strings generated by JavaScript.
- Closed design composition and product interaction styling.

Forbidden responsibilities:
- Generic Atome framework style rules that must be reusable outside eVe.

## Explicit Open / Closed Design Contract

Design ownership follows product specificity:
- Atome open design may contain only product-neutral Squirrel component structure, reusable framework visual assets, shell-level CSS, vendor CSS, and generated framework artifacts.
- eVe closed design owns product chrome, panels, menu, ribbon, flower, Matrix, Finder-facing product UI, private tool visuals, Molecule/MTraX visuals, branding, and product-specific Atome object rendering.
- Static CSS files are not the source of truth for eVe product design. eVe product visuals must be driven by JavaScript tokens, factories, structured style objects, and documented generated style exceptions.
- Generated or vendored CSS may be referenced as an exception only when this map records its ownership and purpose.

Cross-boundary design rules:
- Atome modules must not import or depend on eVe product tokens, panel chrome, product factories, branding, or closed style generators.
- eVe design modules may consume Atome product-neutral assets or Squirrel component contracts, but must not redefine generic framework styling as closed product-only behavior.
- Product object styling for Atome instances inside eVe remains closed design behavior unless a reusable rendering contract is explicitly promoted to Atome.
- Any promotion of eVe visual behavior into Atome requires a product-neutral API, reusable token contract, tests or visual verification, and updates to `maps/CODEMAP.md`, `maps/API_MAP.md`, and `maps/ARCHITECTURE_MAP.md`.

## Core Token Sources

### System UI Tokens

Primary source: `eVe/elements/system_ui_tokens.js`

Role:
- Root source for product-neutral-looking UI values used by eVe surfaces.
- Defines theme colors, CSS variable names, input tokens, tool metrics, panel chrome metrics, shadows, blur, and close button values.

Exports verified:
- `SYSTEM_UI_THEME_TOKENS`
- `SYSTEM_UI_ROOT_VARS`
- `SYSTEM_UI_METRICS`
- `SYSTEM_UI_INPUT_TOKENS`
- `SYSTEM_UI_INPUT_METRICS`
- `SYSTEM_UI_PANEL_CHROME_TOKENS`
- `SYSTEM_UI_PANEL_CHROME_METRICS`

Design rule: new eVe visual modules should use these tokens before inventing local colors, shadows, radii, or input metrics.

### eVe Look and Base Presets

Primary source: `eVe/elements/eVe_look.js`

Related sources:
- `eVe/elements/look/css_preset.js`
- `eVe/elements/look/preset_ensurers.js`

Role:
- Translates system tokens into `eveTokens`, `eveCssVars`, and `evePresets`.
- Installs root CSS custom properties through `applyEveCssVars()`.
- Injects base product rules through `ensureEveBaseStyle()`.
- Owns base rules for inputs, placeholders, finder rows, scope prompts, and shared product UI affordances.

Design rule: broad product styling belongs here only when it is genuinely shared. Feature-specific styling belongs in the feature visual module.

### Panel Chrome Tokens

Primary source: `eVe/elements/design/panel_chrome_tokens.js`

Related sources:
- `eVe/elements/design/panel_chrome.js`
- `eVe/elements/design/dialog_runtime.js`
- `eVe/elements/design/dialog_bounds.js`
- `eVe/elements/design/panel_overflow_indicators.js`
- `eVe/elements/design/panel_fullscreen_runtime.js`

Role:
- Defines the shared panel header/footer/tools dock chrome.
- Converts system panel chrome values into eVe CSS variables and style objects.
- Supports the current panel contract: header, body, tools dock, footer, close control, resize grip, and overflow indicators.

Design rule: all eVe panel chrome changes must route through this layer or `eVe/elements/design.js`, not through isolated panel-local clones.

## Design Factories and Runtime Surfaces

### eVe Element Factory

Primary source: `eVe/elements/design.js`

Role:
- Main closed product factory for dialogs, rows, text, buttons, inputs, editable text, and panel composition.
- Applies eVe CSS variables and base styles during module load.
- Creates panel DOM and style objects through JavaScript, using `evePresets` and panel chrome helpers.

Boundary status: Closed product design API.

### Intuition Layers and Layout Runtime

Primary sources:
- `eVe/intuition/runtime/layer_contract.js`
- `eVe/intuition/runtime/panel_layout_policy.js`
- `eVe/intuition/runtime/panel_api.js`
- `eVe/intuition/core/layers.js`

Role:
- Owns visual stacking, layer targets, panel layer contracts, and z-index semantics.
- Design changes that move UI between layers must be checked against these contracts.

Design rule: z-index values should come from layer contracts or visual tokens, not one-off literals.

### Tool Visual System

Primary sources:
- `eVe/intuition/tools/visual/tool_visual_tokens.js`
- `eVe/intuition/tools/ui/tool_button_factory.js`
- `eVe/intuition/tools/core/tool_definition_ssot.js`
- `eVe/intuition/tools/core/tool_runtime.js`
- `eVe/intuition/projection/button.js`
- `eVe/intuition/projection/tool_strip.js`

Role:
- Owns canonical tool shape, dimensions, active/latched state colors, icon behavior, inline input visuals, projections, and editor-related visual tokens.
- Product tools should share this runtime instead of duplicating button CSS or DOM structure.

Design rule: a business capability has one visual tool identity across toolbox, projection, panel, Finder, and MCP-triggered contexts.

### Menu, Ribbon, Flower, Matrix

Primary sources:
- `eVe/intuition/ribbon/tokens.js`
- `eVe/intuition/menu/visual/toolbox_styles.js`
- `eVe/intuition/menu/visual/toolbox_runtime_visual.js`
- `eVe/intuition/flower/menu.js`
- `eVe/intuition/tools/contextual/flower_menu_visual.js`
- `eVe/intuition/matrix/visual/matrix_visual_tokens.js`
- `eVe/intuition/matrix/ui/view.js`

Role:
- Ribbon tokens define handle icons, tool sizes, flower metrics, drag thresholds, and animation timing.
- Toolbox styles inject runtime CSS variables and rules for menu V2.
- Matrix visual tokens define grid metrics, overlay styling, tile states, labels, and open animation.

Design rule: menu/ribbon/flower/Matrix changes must preserve shared tool visual semantics and layer ordering.

### Panel and Component Visuals

Primary sources:
- `eVe/intuition/panels/visual/panel_visual_tokens.js`
- `eVe/intuition/panels/core/panel_creator.js`
- `eVe/intuition/components/visual/component_visual_tokens.js`
- `eVe/intuition/components/core/component_creator.js`

Role:
- Owns panel surfaces, panel layer style, component tokens, and reusable product component visuals.

Design rule: new panels should use the panel creator and shared panel chrome before adding panel-local styling.

### Atome Visual Objects

### MTraX and Molecule Visuals

Design rule: MTraX/Molecule visuals are closed product workflow design unless promoted through a deliberate Atome media contract.

## Static CSS and Generated CSS Exceptions

Static CSS files currently found:
- `atome/src/css/squirrel.css`
- `atome/src/js/event-calendar.css`
- `atome/src/js/leaflet.min.css`
- `dist/squirrel.css`
- `node_modules/leaflet/dist/leaflet.css`

Ownership:
- `atome/src/css/squirrel.css`: framework/browser shell CSS.
- `atome/src/js/event-calendar.css` and `atome/src/js/leaflet.min.css`: bundled third-party or library CSS.
- `dist/squirrel.css`: generated distribution artifact.
- `node_modules/leaflet/dist/leaflet.css`: dependency artifact.

Rule: these files are not the source of truth for eVe product design. Do not edit `dist/` or `node_modules/` manually.

## Asset Surface

Primary paths:
- `atome/src/assets/images/icons/`
- `atome/src/assets/images/logos/`
- `atome/src/assets/images/`

Role:
- Shared icon and logo source for Atome/eVe runtime UI.
- Icons are consumed by JavaScript factories through image tags, masks, filters, and bundled asset URL normalization.

Design rule: icon styling belongs in visual factories or tokens. Do not duplicate SVG variants just to express state when color/filter/mask tokens can express it.

## Validation Expectations

For any design change:
- Verify the owning JavaScript visual module directly.
- Check whether the change should be tokenized in `system_ui_tokens.js`, `eVe_look.js`, or a feature visual token module.
- Verify touched file size and avoid expanding oversized visual/runtime files.
- Run the narrowest relevant visual/runtime test when one exists.
- For significant UI changes, use browser or probe verification on the real rendered surface.

## Known Gaps

- This map is not yet an exhaustive selector/token inventory.
- Some legacy components under `atome/src/squirrel/components/` still contain hardcoded colors and French comments; they require later cleanup before being treated as polished open design contracts.
- Some feature modules inject local `<style>` strings, such as capture and MTraX. These should remain feature-owned unless a repeated pattern justifies promotion into shared tokens.
- A future architecture map should reference this file as the design authority for JavaScript-generated UI styling.
